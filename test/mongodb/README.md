# MongoDB Integration Suite and Fixtures

These JSON files contain a frozen dataset generated by `test/mongodb/cmd/generate-fixtures/main.go`.

## Purpose

They represent the **production-shape contract** — the BSON document shapes
that the current MongoDB driver writes. During a driver upgrade, the
integration tests restore these files into a clean MongoDB and verify that
the new driver can still read them correctly through the repository interfaces.

## How they were generated

```bash
./test/mongodb/scripts/generate-fixtures.sh
```

The generator connects to the docker-compose MongoDB, writes documents through
the repository interfaces (same code path as production), exports the raw BSON
as Extended JSON, and writes the files here.

## How to run the integration suite

From repository root:

```bash
make test-integration-db
```

Equivalent script:

```bash
./test/mongodb/scripts/run-tests.sh
```

The script starts MongoDB via Docker Compose, runs `go test -tags integration ./test/mongodb/...`,
and always tears Mongo down on exit.

## Driver version checks (v1 and v2)

This suite is intended to stay stable while you switch Mongo driver versions in LPS.

### Run with v1 driver (current default line)

1. Ensure `go.mod` uses v1, for example:
   - `go.mongodb.org/mongo-driver v1.17.4`
2. Run:

```bash
make test-integration-db
```

### Run with v2 driver (upgrade validation line)

1. Switch LPS imports and module to v2.
2. Use a v2 version in `go.mod`, for example:
   - `go.mongodb.org/mongo-driver/v2 v2.0.0` (initial migration check), or
   - `go.mongodb.org/mongo-driver/v2 v2.5.0` (latest stable revalidation used in this task)
3. Run:

```bash
make test-integration-db
```

If tests fail after version switch, first check for mixed v1/v2 imports in `test/mongodb`
and in Mongo adapter code.

## When to regenerate

- **Do NOT** regenerate as part of normal test runs
- Only regenerate when intentionally updating the compatibility contract
  (e.g. new fields, new collection, schema migration)
- After regenerating, review the diff carefully before committing

## Format

Each file is a JSON array of BSON documents (with `_id` fields removed for
portability). The test restore helper decodes them as `bson.D` and inserts
via `InsertMany`.
